# GridLens — Build Plan & Specification (v2)

*An open-source GB electricity analytics agent, built to industry standards: a typed, tested Python engine, operated by a CLI and served by a FastAPI service, packaged as a Claude skill/plugin, rendering a self-contained dashboard.*

This is the canonical brief for the build. It is written to be handed to **Claude Code** as the source of truth, and for **Hira** to read and follow. It states the reasoning behind every major decision, not just the instruction, so each choice can be defended in an interview and adjusted sensibly if requirements change.

**Decisions locked in this version:** Chart.js (vendored locally) as the default chart renderer; full industry-standard tooling (uv + pyproject, Pydantic, ruff, mypy, pre-commit, pytest, GitHub Actions, logging); **architecture tier = engine package + Typer CLI + FastAPI service + Dockerfile + static dashboard on GitHub Pages**, with a live-hosted API treated as optional; Claude plugin published with a self-hosted marketplace.

> **Working name:** `gridlens` (rename freely before scaffolding so the package, CLI, plugin, and skill names stay consistent).

---

## 1. Why this project exists

The goal is a **public, linkable portfolio piece** that demonstrates the engineering discipline of a private client system on open data anyone can inspect. The private system cannot be shown (NDA-bound client work); this is its verifiable public twin — a hiring manager can clone the repo, read the code, run it, hit the API, and view a working dashboard.

It is built deliberately to **industry standards** because the project's job is to prove the author can work on real projects — so the engineering practices (typing, validation at boundaries, tests, reproducible environment, CI, a real service layer) are part of the deliverable, not incidental polish. The rare, hireable signal at the centre is **trustworthy numbers from an LLM-driven system**: deterministic code does the maths, every figure is validated against a known reference, the system fails loudly instead of inventing values, and the whole thing is packaged and served the way a real team would ship it.

**Definition of done (this tier):**
- A public GitHub repo with permissive licence, README, and **green CI** (lint, type-check, tests).
- The engine as an installable, typed, tested Python package (`pyproject.toml` + `uv` lockfile).
- A **Typer CLI** to operate it and a **FastAPI** service exposing the metrics as JSON (with auto-generated OpenAPI docs), plus a **Dockerfile** for a reproducible run.
- A self-contained HTML dashboard (Chart.js vendored locally) generated by the engine and hosted free on **GitHub Pages**.
- A small set of Claude skills packaged as a plugin, installable via a self-hosted marketplace.
- A validation report showing the engine reconciles to the data source's own figures to a stated tolerance across every period.
- A README that frames the work honestly and links the live dashboard.

## 2. Guiding principles (the design DNA)

Non-negotiable; they shape every later decision.

1. **Code does the maths.** Every metric, aggregate, and comparison is computed in deterministic Python. The model never does arithmetic or date counting. *Why:* it is the one decision that makes output trustworthy and reproducible.
2. **The data source is the reference; reconcile to it.** Where the API publishes its own aggregate, the engine recomputes it independently and reconciles. *Why:* trustworthiness is proven against a known-good reference, not asserted.
3. **Validate every period to a stated tolerance.** Validation runs across every half-hour / day in the window and reports a match rate. *Why:* one matching example proves nothing.
4. **Fail loud, never fabricate.** When a value cannot be computed (missing data, unexpected schema, API error), return a clear "could not compute" / dash — never a guess. *Why:* a plausible-looking fabricated number is worse than an honest gap.
5. **Config-driven, not hardcoded.** Region selection, fuel-factor mapping, labels, and thresholds live in validated config. Adding a region requires no code change. *Why:* it demonstrates a generalisable system, mirroring "add a client without touching code."
6. **Definitions are fetched, not invented.** Carbon-intensity factors are pulled live from the API's factors endpoint, not hardcoded. *Why:* the API owns the definitions; hardcoding risks silent drift.
7. **Clean room, properly licensed.** All code written from scratch; no client code, data, metric definitions, attribution factors, or validation figures appear anywhere; data is openly licensed and attributed; the repo carries a permissive licence. *Why:* keeps the project unambiguously the author's own and safe to publish (see §16).
8. **Built like production software.** Typed (`mypy`), validated at the boundary (`pydantic`), tested (`pytest` + coverage), reproducible (`uv` lockfile + `Docker`), with automated checks (CI + `pre-commit`). *Why:* the project must demonstrate real-project readiness, so the practices are the point.

## 3. What it does (product definition)

GridLens answers, for the GB electricity grid, the same family of questions the private system answered for marketing performance:

- *"How is the grid doing now / today / this week?"* — current and period carbon intensity, the generation mix, and how today compares to yesterday and last week.
- *"Is anything unusual?"* — deterministic anomaly flags (intensity unusually high/low vs the recent norm, a record-clean or record-dirty period, an abnormal swing in a fuel's share).
- *"Show me a clean dashboard."* — a self-contained HTML dashboard with KPI tiles, the fuel mix, an intensity trend, a period comparison, an anomaly section, and a methodology/validation footer.

It is operated three ways over one engine: a **CLI** (run a report, validate, launch the service), a **FastAPI** endpoint (metrics as JSON), and **conversationally through Claude** via the skills. The dashboard is generated by the engine from the same computed metrics the API serves.

## 4. Data sources

### 4.1 Primary — Carbon Intensity API (official, NESO)

- **Base URL:** `https://api.carbonintensity.org.uk`
- **Authentication:** none — no key, no sign-up.
- **Licence:** CC BY 4.0 — **must attribute NESO** (see below).
- **Times:** all UTC; handle BST/GMT explicitly for any "day" boundary.
- **Granularity:** half-hourly (settlement periods).

**Endpoints used (confirm exact field names against live responses when building):**

| Purpose | Endpoint |
|---|---|
| Current half-hour intensity | `GET /intensity` |
| Intensity for today / a date | `GET /intensity/date`, `GET /intensity/date/{date}` |
| Intensity over a range | `GET /intensity/{from}/{to}` (also `/pt24h`, `/fw24h`, `/fw48h`) |
| **Published carbon factors per fuel** | `GET /intensity/factors` |
| **Statistics (max/avg/min)** | `GET /intensity/stats/{from}/{to}` (and `/{block}`) |
| **Generation mix** | `GET /generation`, `GET /generation/{from}/{to}`, `GET /generation/{from}/pt24h` |
| Regional intensity | `GET /regional`, `/regional/england|scotland|wales`, `/regional/regionid/{regionid}`, `/regional/postcode/{postcode}` + time-range variants |

**Key response shapes (verify live):**
- *Intensity* item: `{ from, to, intensity: { forecast, actual, index } }`, `index` ∈ low/moderate/high/...
- *Statistics* item: `{ from, to, intensity: { max, average, min, index } }`.
- *Generation* item: `{ from, to, generationmix: [ { fuel, perc }, ... ] }`; expect fuels `biomass, coal, imports, gas, nuclear, other, hydro, solar, wind`.
- *Factors*: an object mapping fuel-type names → gCO₂/kWh. **The factor fuel names are more granular than the mix fuel names** (gas split combined/open cycle, imports split by interconnector); a mapping step is required (§8) and the mismatch must be documented honestly.

**Etiquette:** the API is generous, but cache responses (handled by `requests-cache`); one fetch per report is plenty for a snapshot.

### 4.2 Secondary / optional — NESO Data Portal (CKAN)
Historical demand and generation beyond the Carbon Intensity API. CKAN Datastore API; **limit to ~2 requests/minute**; open-licensed. Not required for this tier (see §18).

### 4.3 Optional enhancement — Octopus Energy Agile prices
A public pricing feed for the Agile tariff, commonly keyless for product/tariff reads, enabling a "cheapest & greenest window" feature. **Verify the current endpoint and terms before use.** Stretch only.

### 4.4 Attribution (required)
Because the API is CC BY 4.0, credit the source in the README, the dashboard footer, and a `NOTICE`/`DATA_LICENCE.md`:
> *Carbon intensity and generation data © National Energy System Operator (NESO), via the Carbon Intensity API (carbonintensity.org.uk), used under CC BY 4.0.*

| Type checking | `mypy` | standard static type checker for typed Python |

Knowing when **not** to abstract (YAGNI) is itself an industry signal; the SVG path is a documented future option (§18), not built now.

## 6. Architecture

A clean core engine with thin operation layers on top — the standard "library + interfaces" shape.

**Core engine (`src/gridlens/`, framework-agnostic, pure + typed):**

| Module | Role | Private-system analogue |
|---|---|---|
| `client.py` | HTTP fetch with timeouts, retry/backoff, `requests-cache`; raises custom errors | data-access layer |
| `models.py` | Pydantic models for API responses, metrics, anomalies | typed schema |
| `config.py` | Pydantic settings + profile loading/validation | per-client profiles |
| `metrics.py` | All deterministic metric calculations | the calculation engine |
| `validation.py` | Two-layer reconciliation against the API's own figures | the every-period tolerance check |
| `anomalies.py` | Deterministic, rule-based flags | rule-based anomaly flagging |
| `render.py` | Jinja2 + vendored Chart.js → self-contained HTML | the dashboard renderer |
| `exceptions.py` | Custom exception hierarchy (`GridLensError`, `DataUnavailableError`, …) | fail-loud foundation |
| `logging_config.py` | Logging setup | — |

**Operation layers (thin, all call the core):**
- `cli.py` — **Typer** CLI (§12).
- `api.py` — **FastAPI** app (§11).
- `render.py` — dashboard generation (§10), callable from CLI or API.

**Claude skills layer (4 skills)** that drive the engine via its CLI (§13).

**Packaging:** the engine is a normal pip-installable package; the plugin bundles the skills; the skills invoke the installed `gridlens` CLI.

**Config-driven regions:** a profile is a small, Pydantic-validated YAML file describing a "view":
```yaml
name: Great Britain
scope: national
region_id: null
endpoints:
  intensity: /intensity
  generation: /generation
  stats: /intensity/stats
labels:
  title: Great Britain electricity
```
A regional profile sets `scope: regional` and a `region_id`. Adding "South Scotland" = adding a profile file — no code change. This is the headline "generalisable, not a one-off" property; make it real and test it (§15).

## 7. Metrics (all computed in code, typed, value-or-explicit-NA)

Compute deterministically from the half-hourly series using `pandas`.

- **Carbon intensity over the window:** mean, min, max (gCO₂/kWh), and the timestamps of the cleanest and dirtiest half-hour. Use `actual` where present, else `forecast` (and label which).
- **Index band distribution:** share of half-hours in each band.
- **Generation mix (window average):** mean `perc` per fuel; rank fuels by share.
- **Low-carbon vs fossil split:** low-carbon = wind + solar + hydro + nuclear + biomass\*; fossil = gas + coal; imports and other shown separately. \**Treating biomass as low-carbon is a modelling choice — state it in the footer; it is debated.*
- **Renewable share:** wind + solar + hydro (+ optionally biomass, stated).
- **Period-over-period deltas:** today vs yesterday, this 7-day window vs the previous — absolute and percentage change for intensity and each fuel's share.
- **Records within the window:** lowest/highest single-period intensity; highest renewable share.

Every metric returns a typed value or an explicit "not available"; none silently defaults to zero.

## 8. Validation design (the crux — feature it in the README)

Two layers, deliberately different tolerances and purposes.

### Layer B — Exact aggregate reconciliation *(rigorous, tight)*
Recompute the window's mean/min/max intensity from the half-hourly `/intensity` series, then reconcile against `/intensity/stats/{from}/{to}` for the same window.
- **Expectation:** near-exact match — both derive from the same series; only difference is rounding (stats returns integer gCO₂/kWh).
- **Tolerance:** tight — within ±1 gCO₂/kWh or < 1% relative. Report the match rate across every window tested.
- **Purpose:** proves the engine's arithmetic and date handling against a known-good reference.

### Layer A — Independent intensity reconstruction *(honest, looser)*
Reconstruct an indicative intensity from the mix and published factors: `intensity_reconstructed = Σ(fuel_fraction × factor)`, pulling factors live from `/intensity/factors` and mapping the granular factor names onto the mix fuel names. Compare to reported `/intensity` actual per half-hour.
- **Expectation: this will NOT match exactly, and that is the point.** The official figure uses a fuller methodology (interconnector imports by source, T&D losses, embedded wind/solar, demand weighting), so a naive mix×factor reconstruction is *indicative*.
- **Tolerance:** looser; do not claim a tight match. **Report the distribution of differences** (mean, spread) and **flag outlier periods** where the gap is unusually large.
- **Purpose:** demonstrates the engine genuinely computes independently, quantifies the gap honestly, and surfaces interesting periods.

**Output:** a `validation_report` (emitted by the CLI/API and saved as `examples/validation_report.md`) stating, per window: Layer B match rate at the tight tolerance, and Layer A difference distribution with flagged outliers. The dashboard footer summarises it in one line and links the full report. Layer B also runs as an **automated test** against fixtures (§15).

## 9. Anomaly rules (deterministic, simple, honest)

Two to three rule-based flags (no ML), thresholds in config, each framed as an **observation to verify** and carrying the numbers behind it.
1. **Intensity deviation:** latest half-hour > X% (default 15%) from the rolling N-day (default 7) average for the same period-of-day.
2. **Record clean/dirty period:** a half-hour sets a window low/high for intensity, or renewable share sets a window high.
3. **Fuel-share swing:** a fuel's share moves more than Y percentage points (default 15pp) vs its recent average.

## 10. The dashboard

A **self-contained HTML file** — all CSS inline, **Chart.js vendored locally** (commit `chart.umd.js` under `static/` and reference it by relative path, never a CDN), data baked in at render time. It works on GitHub Pages with no runtime API calls and no internet.

**Before any UI work, the builder must read the `frontend-design` skill** (`/mnt/skills/public/frontend-design/SKILL.md`) and follow its tokens and constraints. No browser storage APIs.

**Accessibility (Chart.js renders to `<canvas>`, a single bitmap):** every chart canvas needs `role="img"`, a descriptive `aria-label`, and text fallback between the tags. This is the one trade-off vs SVG (canvas isn't screen-reader-native and doesn't scale as crisply) and is mitigated by these attributes.

**Sections (outcomes-first):**
- **Freshness line** — window covered, "data as of …", actual vs forecast.
- **KPI tiles** — average intensity (index band as a coloured badge), renewable share, low-carbon vs fossil split, cleanest/dirtiest period; each with its week-over-week change and a small sparkline.
- **Generation mix** — a donut or horizontal bar of the window-average mix.
- **Intensity trend** — the half-hourly / daily series.
- **Period comparison** — this window vs previous, per metric.
- **Anomaly section** — flags with supporting numbers, framed as observations.
- **Methodology & validation footer** — modelling choices, Layer B match rate, the Layer A note, and the NESO CC BY 4.0 attribution.

The dashboard consumes the **same computed metrics the API serves** (single source of truth). Use an intuitive intensity colour ramp (green = clean → red = dirty), checked for colour-blind distinguishability.

## 11. The API (FastAPI)

A thin service over the engine — the strongest "I can serve an engine" signal, and small because it only calls the core.

**Endpoints (responses are Pydantic models reused from the engine):**
- `GET /health` → liveness.
- `GET /metrics?profile=gb&from=…&to=…` → computed metrics JSON.
- `GET /anomalies?profile=gb&…` → flags.
- `GET /validation?profile=gb&…` → validation report JSON.
- `GET /dashboard?profile=gb&…` → rendered HTML (optional convenience).

**Practices:** run with `uvicorn`; **OpenAPI/Swagger docs auto-served at `/docs`** (a free, real-feeling deliverable); map custom exceptions to proper HTTP status codes (e.g. missing data → 404, bad params → 422, upstream failure → 502); request timeouts throughout. Keep all computation in the engine — the API just orchestrates and serialises.

## 12. The CLI (Typer)

Operate the tool like a real data utility:
- `gridlens report --profile gb --days 7 --out examples/gb.html` — fetch, compute, render the dashboard.
- `gridlens validate --profile gb --days 7` — run both validation layers, print/save the report.
- `gridlens fetch --profile gb --days 7` — fetch + cache raw data (debugging).
- `gridlens serve --port 8000` — launch the FastAPI service.

Typed options, `--help` for every command, clear error messages from the exception hierarchy.

**Packaging decisions (and why):**
- **`src/` layout, pip-installable package.** *Why:* the standard, import-safe Python package shape; the CLI and API import the same engine, so logic lives once.
- **Config in `pyproject.toml`** for ruff, mypy, and pytest. *Why:* one place; the modern convention (no scattered `.flake8`/`setup.cfg`).
- **Profiles bundled in the package (`profiles/`), validated by Pydantic.** *Why:* needed at runtime; validation catches a bad profile loudly.
- **Chart.js vendored in `static/`.** *Why:* offline, self-contained dashboard; no CDN dependency at view time.
- **Skills are thin wrappers that call the installed `gridlens` CLI.** *Why:* cleaner and more realistic than copying scripts into each skill; the skill's `requirements.txt` installs the package, and the plugin bundles the SKILL.md files. *(If a fully self-contained plugin is wanted, the package can also be bundled and referenced via `${CLAUDE_PLUGIN_ROOT}` — but the CLI-call approach is preferred.)*
- **Tests in the repo, excluded from any packaged artifact.** *Why:* tests are for CI/review, not runtime.
- **Plugin + marketplace versions kept in sync**, starting at `0.1.0`.
